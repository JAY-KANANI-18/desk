# Shared UI Transformation Status

Any AI agent touching `frontend_new` must read `00_READ_THIS_FIRST_FRONTEND_RULES.md`, `01_SHARED_UI_TRANSFORMATION_STATUS.md`, `02_UI_AUDIT_BASELINE.md`, and `03_UI_TRANSFORMATION_CHECKLIST.md` before making UI changes.

## Goal

We are gradually replacing one-off UI controls with the shared `src/components/ui/` layer without redesigning the product. The priority is consistency, reusability, and stability. The rule is:

- prefer shared variants and props
- avoid page-level styling hacks on shared components
- preserve current visual identity and behavior

## Current implementation rules

- Before changing shared component defaults, variant styling, or behavior, search all existing consumers of the affected component/variant/prop and verify the blast radius. Prefer additive, default-preserving props unless every affected screen is intentionally updated.
- Shared components should absorb visual variations through props like `appearance`, `labelVariant`, `radius`, `variant`, and `loadingMode`.
- Shared `VerificationCodeInput` should own multi-cell verification and OTP entry flows instead of page-local raw digit inputs.
- Shared `DisclosureButton` can now carry a warning tone for cautionary or demo-credential expanders instead of route-local raw disclosure buttons.
- Shared modal header/footer structure now lives in `src/components/ui/modal/`: use shared modal props for `onBack`, `headerIcon`, `footerMeta`, `secondaryAction`, and `primaryAction` so CTA placement stays standardized.
- Shared modal shells now also support contextual drawers through props like `showOverlay`, `allowBackgroundInteraction`, and `lockBodyScroll` when a side panel should stay visually integrated with the page instead of dimming the whole screen.
- Shared `PageLayout` now supports an optional leading control for icon-only back navigation on desktop subpages; keep title areas concise and avoid descriptive subtitle text on standardized workspace/channel/workflow subpage headers unless the screen truly needs it.
- Shared `PageLayout` keeps desktop title content on its own row and renders search, filters, badges, and action buttons in a consistent secondary header row beneath it.
- The root compatibility wrapper at `src/components/ui/Tooltip.tsx` should always re-export the new portal/mobile-aware tooltip implementation under `src/components/ui/tooltip/`.
- Shared `Input` now supports scheduling-friendly input types like `date`, `time`, and `datetime-local` so live forms do not fall back to one-off date-time fields.
- Shared `Input` also supports a compact `composer-inline` appearance for email header rows like subject, cc, and bcc so live composers do not fall back to raw inline inputs.
- Shared `BaseInput` supports `hideNativePickerIndicator` for date/time fields with shared picker adornment buttons, so pages do not need local browser-selector classes.
- Shared `Button` now supports left-aligned full-width action rows through `contentAlign="start"` so dropdown and sheet action lists do not need one-off raw buttons just for alignment.
- Shared `Button` also supports compact `2xs`/`xs` actions plus `dark` and `danger-ghost` variants so tight icon rows, dark publish actions, and subtle destructive actions do not fall back to one-off styling.
- Shared `Button` supports an `inverse-primary` variant for white CTAs on primary/dark promotional cards without route-local button styling.
- Shared `Button` supports a branded `facebook` variant for Meta/Facebook OAuth CTAs without route-local button color styling.
- Shared `Button` supports a `dashed` variant for add-another/drop-zone style actions like lifecycle stage creation without page-local dashed button markup.
- Shared `IconButton` mirrors the shared button radius API so icon-only header/back/import controls can keep standardized rounding without local shape classes.
- Shared `Button` also supports `soft`, `soft-primary`, `soft-warning`, and `unstyled` variants so selected filters/toggles and richer route-specific pressables can stay on shared button semantics without falling back to raw controls.
- Shared `Button` also supports an `inherit-ghost` variant plus `radius="none"` so inline bubble actions and segmented full-width action rows can inherit their surrounding text tone without rebuilding page-local button markup.
- Shared `Button` `unstyled` now removes default padding/min-height for compact inline text-edit triggers that still need shared button semantics.
- Shared `Button` now supports a `list-row` variant for full-width clickable list items with owned row padding, bottom border, selected background, hover behavior, visible overflow for overlapped badges, normal row text wrapping, and top-aligned multi-line content so migrated lists do not depend on `unstyled` for row surfaces.
- Shared `Button` now supports a `tab` variant with a selected bottom border so compact tab rows like inbox Open/Closed can keep a visible active underline without page-local border overrides.
- Shared `Button` also supports a `select-card` variant plus `selected` state so option-card selectors can stay on shared button semantics without reverting to page-local raw card buttons, and it now also supports child-layout preservation plus proper left text alignment for full-width custom content rows; shared `SelectableCard` wraps that behavior for reusable selector rows with the label grouped to the left and a trailing selection indicator pinned at the far edge.
- Shared `DisclosureButton` supports compact plain disclosure headers for dense panel sections without rebuilding accordion controls locally.
- Shared `DisclosureButton` also supports a primary tone for informational disclosure headers like workflow trigger-condition panels without rebuilding local accordion buttons.
- Shared select primitives now support compact `pill`, `inline`, and borderless `toolbar` trigger appearances plus inset toned option rows, and `CompactSelectMenu` wraps those shared pieces for header/tool-pickers that need searchable dropdowns without rebuilding local menu logic. The compact picker also keeps selected-state checks rounded and right-aligned, its trigger heights stay aligned with shared header actions, opening the menu now highlights the actual selected option instead of the first row, and non-selected hovered/highlighted inset rows no longer reuse the selected tinted background.
- Use `WorkspaceTagManager` for full workspace-tag areas that both add/search tags and render the selected tags below the control. This is the default for inbox, broadcast, imports, contacts, and workflow tag surfaces.
- Use `WorkspaceTagSelect` as the lower-level primitive only when a surface genuinely needs custom composition beyond the shared tag-management area. If a surface needs a narrower popover, right-aligned menu, or custom trigger row, extend the shared API rather than rebuilding the picker locally.
- Use `CountBadge` for compact numeric item/unread counts in tabs, sidebar rows, and filter rows instead of stretching `Tag` or rebuilding page-local count pills.
- Use `CheckboxInput` for real checkbox semantics like confirmations and multi-select cards. Do not substitute `ToggleSwitch` where the UI is conceptually a checkbox.
- Shared `CheckboxInput` supports a compact `sm` size for dense panels where checkbox semantics are still required.
- Use `RangeInput` for shared slider semantics instead of page-local raw range inputs.
- Use `ColorInput` for branded color pickers instead of hand-rolled `input type="color"` plus text field pairs.
- Use `TagColorSwatchPicker` for preset workspace-tag color choices instead of page-local swatch rows.
- Use `TagInput` for token/chip entry flows such as allowed websites, domains, and similar list-entry surfaces.
- Shared `TagInput` supports suggestion rows and custom display labels so workflow token selectors can stay on the shared token-entry primitive.
- Shared `BaseInput` supports an `inline-edit` appearance for compact table/list rename fields instead of route-local border-bottom input styling.
- Shared `Avatar` supports a neutral fallback tone, a 40px `base` list avatar size, and a tiny `2xs` size for compact user chips; shared `AvatarWithBadge` supports overlap-positioned image badges so channel overlays can match inbox message/list avatar positioning without page-local badge markup.
- Use `DisclosureButton` for accordion/disclosure headers like danger zones instead of one-off raw buttons.
- Do not add inline `style` or page-specific `className` to shared `Button`, `BaseInput`, `TextareaInput`, or `Select` just to restore a desired look. Add or improve a shared variant instead.
- Use the simpler shared `Select` wrapper for live surfaces where the older native select experience is still preferred.
- Tooltips remain desktop-only.
- Channel live pages under `src/pages/workspace/channels/` and `src/pages/channels/` are now expected to start from shared primitives first; a raw-control scan there should stay clean unless a genuinely new control type is introduced.
